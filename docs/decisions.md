## 플랫폼 지향점 — 최상위 기준 (2026-07-19, 사용자 확정)

- 이 플랫폼은 개인 도구가 아니라 **산하 전 팀원이 함께 쓰는 팀 플랫폼**이다.
- **어떤 기기(PC·태블릿·휴대폰)에서 접속하든 모든 업무가 가능해야 한다.** 열람만이 아니라
  고객 등록·수정·삭제, 일정 등록, 발행까지 전부.
- 개발(코드 작업)은 한 대에서 한다 — 현재 안창민 업무용 노트북. 개발 편의 문제이며 사용과 별개다.
- 판단 기준: 어떤 기능이 "총관리자 PC에서만 된다"면 미완성으로 본다.
- 귀결: 로컬 폴더(File System Access) 방식은 휴대폰·태블릿 브라우저에서 동작하지 않는다.
  따라서 팀 공유 백엔드(2차 공사, NCP+Cloudflare)는 선택 사항이 아니라 지향점 달성의 필수 조건이다.
  1차의 로컬 폴더 모드는 오프라인·개인 백업 수단으로 남긴다.
- CLAUDE.md·AGENTS.md 최상단에 동일 조항 반영.

## 공간 성격 정의 — 브리핑과 케어센터 (2026-07-19, 사용자 확정)

- 브리핑(web/brief/) = **관리·실행실**. 자료 생성 기능을 만들지 않는다. 밖에서 만든 브리핑
  자료(문서+스크립트)를 등록(업로드)하고, 보관·목록 관리하고, 발표자·청중 동기화로 실행만 한다.
- 케어센터(web/care/) = **생성·관리실**. 주간·월간 안창민이 케어센터 공정 안에서 만들어지고
  (발행 데스크가 입구), 발행된 호가 서재에 쌓인다.
- 브리핑 자료 제작 공정: 사용자가 Code 세션과 함께 제작 → brief-doc 형식 산출 →
  공개 가능 자료는 data/brief/에, 고객 자료(보장분석)는 로컬 고객 저장소에 보관 → 브리핑에서 등록·실행.

## 기기 역할 분담 (2026-07-19, 사용자 확정)

- **업무용 노트북 = 주력 1대.** 개발(바이브 코딩)·케어센터 발행·고객관리·보장분석·브리핑 전부 여기서.
  Claude Code·git·Python·고객 저장소·원드라이브가 이 기기에만 있으면 된다.
- **개인(집) 노트북 = 사용 전용.** 마이가디언 웹앱을 브라우저로 쓰는 수준. 개발 도구를 깔지 않는다.
  고객 데이터를 두지 않으므로 고객 자료 동기화 문제가 발생하지 않는다.
- 결과: 고객 저장소 비공개 repo는 ① 백업 ② 총관리자의 팀원 자료 열람 ③ 팀원 각자의 기기 용도로만 쓴다.
  두 기기 사이의 실시간 동기화 부담이 사라진다.
- 전제: 개인 노트북에서 쓰려면 **GitHub Pages 활성화가 필요하다** (저장소 소유자만 가능).
  Pages 전에는 업무용 노트북의 로컬 서버로만 열린다.

## 보장분석 자료 유통 원칙 (2026-07-19, 사용자 확정)

- 보장분석 파일은 고객에게 절대 전송하지 않는다. FC의 영업 자산이며 상담 자리에서만 사용한다.
- 재생 환경은 본인 기기(노트북·태블릿·폰) 한정. 기기 간 이동은 원드라이브(1TB) 동기화 폴더가 담당한다.
- 영상 포함 자료 전부 로컬 재생 — 영상 호스팅(R2·유튜브 등) 공사는 불필요해져 예약 취소.
- v1(마누스 허브)에 있는 기존 영상은 서버가 닫히기 전에 원드라이브로 전량 백업한다 (사용자 실행).
- 브리핑 1차의 로컬 파일 열기·로컬 mp4 재생 설계가 이 전제와 일치한다. 자료 보관 위치로
  원드라이브 동기화 폴더를 권장 표기한다.

## 2차 공사 백엔드 선정 (2026-07-19, 사용자 확정)

- 고객 개인정보 저장소: **네이버 클라우드(NCP) 한국 리전**. 국외 이전 문제를 피하고,
  계정을 이미 보유(네이버 API HUB가 NCP 콘솔)해 신규 가입이 없다.
- 개인정보가 아닌 팀 데이터(공지·일정·발행물 메타·브리핑 동기화·조회 통계): Cloudflare 무료 구간.
- 마누스 클라우드는 조사 후 탈락 — AI 학습 사용 가능성, 미국 리전(국외 이전), 반출 경로 취약.
- 클라우드 선정 5대 확인 항목을 원칙으로 둔다: 저장 리전 / 저장 암호화 / 사업자 접근 범위 /
  AI 학습 사용 여부 / 계약 종료 시 반출·삭제 절차. 하나라도 불명확하면 채택하지 않는다.
- 상세: docs/phase2-server-design.md (착공은 1차 마무리 후, 팀 확장 시점)

## 예약 공사 — 세일즈 코칭 시스템 (2026-07-19, 착공 시점 미정)

- 가상고객 상대 세일즈 연습 + 브리핑 발표에 대한 코칭(실시간 또는 사후 피드백). 외부 API 연동 필요.
- 외부 API 호출·음성 전송이 필요하므로 정적 화면에 넣지 않고 2차 공사(Cloudflare Workers) 묶음으로 예약.
- 브리핑 1차에 연동 고리만 심는다: 발표 기록(시작·종료·페이지 전환 타임스탬프)을 로컬에만 남기는 기능.
- 실제 고객 상담의 실시간 녹음·전송 코칭은 고객 동의·개인정보 문제가 있어 별도 판단 전까지 금지.
  가상고객 연습과 발표 사후 피드백만 대상으로 한다.

## 세션 분업 지도 (2026-07-19, 사용자 승인)

채팅방(세션)은 메뉴가 아니라 **작업 성격**으로 나눈다. 메뉴별 세션 분리는 공용 파일 충돌과
디자인 파편화를 낳으므로 금지한다. 모든 세션은 아래 영역표를 벗어나는 파일을 수정하지 않는다.

| 세션 | 담당 영역 | 금지 사항 |
|---|---|---|
| 앱 메인 (화면 전담) | web/ 전 화면, 공용 자산(platform.css·style.css·platform.js·사이드바) | 데이터 파이프라인 실행 |
| 우리팀 | web/team/, data/team/, 팀 스키마 | 공용 파일 수정 |
| 브리핑 (신설) | web/brief/, data/brief/, brief-doc 스키마 | 공용 파일·타 화면 수정 |
| 터미널 (데이터 전담) | data/, pipeline/ (보장분석 P1~P5 포함) | web/ 수정 |
| 발행 (매주 새 방) | data/care/ 발행물 생성만 | 화면·스타일 수정 |
| 자료 제작 (건별 새 방) | data/brief/ 발표 자료 콘텐츠 (강의·교육) | web/ 화면 수정 |
| Codex | 검수 전용 (codex/* 브랜치) | 시공 |

브리핑은 두 갈래로 나뉜다: **화면·엔진은 브리핑 세션**, **자료 콘텐츠는 자료 제작 세션**.
자료 제작은 건별로 새 방을 판다 (예: 당근 상속·증여 모임 강의). 원고 작성은 웹에서 이어가도
무방하며, 완성 원고를 자료 제작 세션에 넘겨 brief-doc 형식으로 변환·커밋한다.

공용 파일(platform.css, style.css, platform.js, 각 화면의 사이드바 마크업) 수정권은
앱 메인 세션에만 있다. 다른 세션이 공용 수정이 필요하면 handoff.md에 요청을 남긴다.

작업 상황 공유 의무 (모든 세션 공통, 예외 없음):
1. 시작 시 docs/current-status.md, docs/decisions.md, docs/handoff.md를 읽는다
2. 종료 시 handoff.md에 변경 내용·검증 결과·미완료·다른 세션에 넘길 요청을 기록한다
3. 종료 전 반드시 commit·push — push 안 된 작업은 다른 세션에 존재하지 않는 것과 같다
4. 새 결정·규칙 변경은 decisions.md에 즉시 기록한다

## 브리핑 업로드 확장 (2026-07-19, 사용자 요구)

- 로컬 자료 형식 확장: 브리핑 문서(JSON) 외에 PDF·HTML·이미지(여러 장) 업로드 지원. 처리 원칙은 동일 — 브라우저 안에서만, 외부 전송 금지
- 스크립트 별도 업로드: .txt/.md 파일을 자료와 함께 올리면 줄 단독 "---" 구분 순서대로 페이지별 발표자 스크립트가 된다. 문서(JSON) 자체 스크립트보다 사이드카가 우선
- 로컬 자료 전달 방식 변경: sessionStorage·doc 메시지 방송 폐기 → IndexedDB 한 슬롯(새 자료가 이전 자료 대체). 두 창이 같은 기기 저장소를 직접 읽어 자료 내용이 채널에 실리지 않는다 (규약 v1.1: doc 제거, scroll 추가)
- PDF 렌더러(pdf.js 3.11.174)는 저장소에 내장(web/assets/vendor/) — 발표 현장 오프라인 대비
- HTML 자료: 스크립트 실행 차단(sandbox, 정적 렌더만) + 스크롤 동기화. 포인터·페이지 넘김은 이 형식에서 미지원
- 교육 자료 갈래: 강제 분류 없음. 문서·목록의 자유 입력 "분류" 필드(예: 보험상품/보험원리)로 배지 표시만 — 자료가 쌓이면 필터 추가 검토

## 역할·권한 모델 확정 — 설계사 플랫폼 (2026-07-20, 사용자 확정)

- 정체성: 이것은 **설계사(FC) 플랫폼**이다. 안창민은 총괄관리자·개발자·실사용자·팀장을 겸한다.
- 역할은 2계층: **팀원(FC)** / **총관리자**.
- 팀원 권한: 업무 기능 전부 — 자기 고객 등록·수정·삭제, 자료 업로드, 브리핑, 팀 스케줄,
  발행까지. 단, 데이터 범위는 자기 고객으로 한정. 원칙: "쓰는 건 전부, 관리는 오직 관리자".
- 총관리자 전용 권한: 팀원 초대·발급 코드 생성·회수, 전체 팀원 고객 열람, 관리자 지정·회수.
- 관리자는 특정인 하드코딩이 아니라 역할(role)로 설계한다 — **추후 관리자 추가 지정 가능 구조**.
- 개발자 역할(코드·배포·git)은 플랫폼 바깥의 일이며 팀원에게 노출되지 않는다.
- 2차 공사 STEP 1 접근 통제 설계는 이 모델을 기준으로 한다.
