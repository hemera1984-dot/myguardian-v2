## 브리핑 시스템 1차 (2026-07-19)

- 상담·교육·강의 3모드를 한 엔진(web/brief/)으로 통합. 모드는 기기 구성·진입 동선 차이일 뿐 화면·동기화는 동일
- 두 시점 원칙: 고객용/발표자용 파일을 따로 만들지 않는다. 한 문서(brief-doc) 안에 스크립트를 넣고 청중 화면에서는 렌더링하지 않는다
- 1차 운반체 = BroadcastChannel(한 기기 두 창, 서버 불필요 — 순수 정적 헌법 준수). 강의 전부·교육의 빔 사용·한 기기 상담을 커버
- 2차(미착공) = Cloudflare Workers WebSocket 원격 동기화. 규약 계층(메시지 v1)과 운반 계층을 분리해 두어 운반체만 교체한다. 단, 상담 문서(개인정보)는 원격 운반체로 전송 금지
- 페이지 캔버스 1280×800 고정 설계 후 배율 축소 — 포인터 좌표를 비율(0~1)로 보내 두 창 크기가 달라도 일치
- 영상은 로컬 파일(video 태그)만. 청중 창은 음소거 재생(자동재생 정책 대응, 소리는 발표자 기기 담당)
- 상담 문서는 로컬 고객 저장소(mg-clients)에서 파일로 연다. 공개 저장소 저장 금지. 보장분석 파이프라인(P1~P5)이 brief-doc 스키마로 산출물을 생성하기로 함 — 스키마 변경 시 이 문서에 기록
- 사이드바 메뉴: 사용자 승인으로 9번째 메뉴로 추가 (아래 헌법 개정 2차 참조). 표기는 "브리핑" 확정

## 브리핑 업로드 확장 (2026-07-19, 사용자 요구)

- 로컬 자료 형식 확장: 브리핑 문서(JSON) 외에 PDF·HTML·이미지(여러 장) 업로드 지원. 처리 원칙은 동일 — 브라우저 안에서만, 외부 전송 금지
- 스크립트 별도 업로드: .txt/.md 파일을 자료와 함께 올리면 줄 단독 "---" 구분 순서대로 페이지별 발표자 스크립트가 된다. 문서(JSON) 자체 스크립트보다 사이드카가 우선
- 로컬 자료 전달 방식 변경: sessionStorage·doc 메시지 방송 폐기 → IndexedDB 한 슬롯(새 자료가 이전 자료 대체). 두 창이 같은 기기 저장소를 직접 읽어 자료 내용이 채널에 실리지 않는다 (규약 v1.1: doc 제거, scroll 추가)
- PDF 렌더러(pdf.js 3.11.174)는 저장소에 내장(web/assets/vendor/) — 발표 현장 오프라인 대비
- HTML 자료: 스크립트 실행 차단(sandbox, 정적 렌더만) + 스크롤 동기화. 포인터·페이지 넘김은 이 형식에서 미지원
- 교육 자료 갈래: 강제 분류 없음. 문서·목록의 자유 입력 "분류" 필드(예: 보험상품/보험원리)로 배지 표시만 — 자료가 쌓이면 필터 추가 검토
