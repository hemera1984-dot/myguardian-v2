# 2차 공사 설계서 — 팀 공유 백엔드 (초안, 사용자 승인 대기)

작성: 2026-07-19. 착공 전 검토용. 헌법 개정이 선행되어야 한다.

> **개정 예고 (2026-07-20)**: 아래 내용 중 Cloudflare 부분은 폐기됐다 — 전부 NCP(유료)로
> 단일화 (decisions.md "2차 백엔드 개정"). 또한 역할 모델 3.0(직급 트리·B안 열람·조인웍 공유),
> CLOVA OCR 고객 등록 파이프라인이 추가됐다. 착공 전에 이 문서를 그 기준으로 재작성할 것.

## 왜 필요한가

1차(순수 정적 + 로컬 폴더)는 **혼자 쓸 때는 완결**이지만, 팀으로 확장하면 세 곳에서 막힌다.

| 막히는 곳 | 1차의 한계 |
|---|---|
| 팀원 고객 데이터 | 각자 PC 로컬 폴더 + git 동기화. 팀원이 동기화를 누르지 않으면 데이터가 갈라진다. 사람이 늘수록 설치·교육 부담이 비례로 는다 |
| 팀원 발행 (주간 OOO) | 집필·발행 도구가 총관리자 PC에만 있다. 팀원은 자기 이름의 잡지를 낼 수 없다 |
| 팀 공지·일정 | 정적 JSON이라 화면에서 쓸 수 없다. 읽기 전용 |

동기화 배치는 총관리자 1인이 기기 2대를 쓸 때의 임시 수단이지, 팀 운영 방식이 될 수 없다.

## 무엇을 서버로 옮기고, 무엇을 로컬에 남기나

**서버로 (팀이 공유하거나 여러 기기에서 써야 하는 것)**
- 고객 데이터 (프로필·메모·보유보험) — FC별 소유권, 총관리자는 전체 열람
- 팀 공지·일정·출근 현황
- 케어센터 발행 (기사 생성 요청 중계, 발행물 저장)
- 브리핑 기기 간 동기화 (원래 예약돼 있던 항목)

**로컬에 남김 (서버로 보낼 이유가 없는 것)**
- 보장분석 파일·상담 영상 — 본인 기기 전용, 고객 미전송 (2026-07-19 확정)
- 사례·판례 383건, 사전 693건, 케어 발행물 본문 — 공개 데이터. GitHub Pages가 더 빠르고 공짜
- 개인 프로필·카톡 서명 — 브라우저 저장으로 충분

정적 사이트를 서버 앱으로 갈아엎는 것이 아니다. **화면·데이터·스키마는 그대로 두고
저장 경로만 추가한다.** 1차에서 만든 스키마(customer·care-issue·care-body·brief-doc)가
그대로 서버 테이블 규격이 된다.

## 기술 구성 — 데이터 성격으로 두 곳을 나눈다 (2026-07-19 개정)

고객 개인정보는 국외 이전 문제(개인정보보호법)가 걸리므로 **국내 리전**에 둔다.
개인정보가 아닌 팀 데이터는 리전 제약이 없으므로 무료 구간이 넉넉한 곳에 둔다.

| 데이터 | 위치 | 이유 |
|---|---|---|
| 고객 프로필·메모·보유보험 | **네이버 클라우드(NCP) 한국 리전** — 2026-07-19 사용자 채택 | 국외 이전 없음. 계정 이미 보유(네이버 API HUB가 NCP) |
| 팀 공지·일정·출근, 발행물 메타, 브리핑 동기화, 조회 통계 | **Cloudflare Workers + D1/R2** | 개인정보 아님. 무료 구간으로 충분 |
| 인증 | GitHub OAuth 또는 FC별 발급 코드 | 자체 회원가입·비밀번호 저장을 만들지 않는다 |
| 화면 | 지금 그대로 (GitHub Pages) | 저장 호출만 fetch로 추가 |

### NCP 사용 전 준비 (착공 시)

- 계정은 이미 있다(네이버 API HUB가 NCP 콘솔). 신규 가입·본인확인 불필요.
- 단, **검색 API 키(X-NCP-APIGW-API-KEY)는 검색 전용**이며 서버·DB 접근 자격이 아니다.
  리소스 생성에는 결제수단 등록과 계정 Access Key가 별도로 필요하다.
- 후보 상품:
  - Micro Server — 결제수단 등록일부터 1년 무료, 계정당 1대, 한국 리전 전용, 성능 보장 없음.
    검증·시험 운영에 적합. 실운영 안정화 시 유료 인스턴스로 전환.
  - Cloud Functions — 요청 수·실행 시간 종량제. 서버 관리가 없어 간헐적 저장/조회 패턴에 적합.
  - Cloud DB for MySQL — 관리형. 고가용성 2인스턴스 구성이라 상대적으로 비용이 크다.
- 실요금은 착공 시 요금 계산기(https://www.ncloud.com/charge/calc/ko)로 산정한다.

### 비용

- Cloudflare 측(개인정보 없는 데이터): 팀 10~20명은 무료 구간 안.
- NCP 측(고객 데이터): 첫 1년은 Micro Server로 사실상 무료 검증 가능. 이후 실산정.
- v1(마누스 풀스택, 천만원)과 성격이 다르다 — 화면을 다시 짓지 않고 저장 경로만 추가한다.

### 검토했으나 탈락한 선택지 — 마누스 클라우드 (2026-07-19)

사용자가 이미 보유한 마누스 클라우드를 백엔드 후보로 조사했다. 탈락 사유:

1. **AI 학습 사용 가능성** — 개인정보 처리방침상 입력 데이터가 모델 학습에 쓰일 수 있다.
   고객 상담 정보를 다루는 이상 단독 탈락 사유다.
2. **미국 리전** — 고객 개인정보 국외 이전에 해당해 정보주체 동의가 필요하다. 현실성 없음.
3. **탈출 경로 취약** — DB 전체 내보내기가 기본 기능이 아니고, 구독 해지 시 삭제되며 유예 기간 명시가 없다.
4. 직접 작성한 서버 코드를 상시 운영하는 PaaS 형태가 아니라 생성 환경에 맞춰야 한다.

마누스 자체 보고서도 "국내 클라우드로 내보내 운영" 을 권고했다. 프로토타이핑 도구로는 유효하나
실운영 백엔드로는 채택하지 않는다.

## 단계

- **STEP 1 — 고객 데이터 API**: FC별 소유권·총관리자 전체 열람. 고객관리 화면의 저장 경로를
  로컬 폴더 → 서버로 전환(로컬 폴더 모드도 남겨 오프라인 대비). 팀 확장의 병목을 먼저 푼다.
- **STEP 2 — 팀 공지·일정 쓰기**: 우리팀 화면에서 팀원이 직접 등록.
- **STEP 3 — 케어 발행 중계**: 화면에서 주제 선택 → 생성 → 초안 편집 → 발행.
  네이버 API·Claude API 키를 서버에 두고 브라우저에 노출하지 않는다.
- **STEP 4 — 브리핑 기기 간 동기화**: 고객 태블릿·수강생 폰. 1차의 메시지 규격을 그대로 사용.

## 헌법 개정안 (조문 초안)

현행:
> - 상시 서버·데이터베이스·로그인 기능 금지.
> - 예외: 브리핑 실시간 동기화는 추후 Cloudflare Workers 엣지 함수로 구현한다.

개정안:
> - 화면과 공개 데이터는 순수 정적을 유지한다 (HTML·CSS·JS·정적 JSON).
> - 팀 공유가 필요한 기능(고객 데이터·팀 공지·일정·발행·브리핑 동기화)에 한해
>   Cloudflare Workers 엣지 함수와 관리형 데이터 저장소 사용을 허용한다.
> - 자체 회원가입·비밀번호 저장은 여전히 금지한다. 인증은 GitHub 계정 또는 FC별 발급 코드로 대신한다.
> - 서버는 저장·중계만 담당한다. 판례·사례 생성 금지 등 데이터 신뢰성 원칙은 서버에도 동일 적용한다.
> - 고객 개인정보는 공개 저장소·공개 경로에 절대 두지 않는다 (서버 저장 시에도 접근 통제 필수).

## 착공 전 확인 사항

1. **개인정보 처리** — 고객 실명·연락처를 외부 클라우드에 위탁 보관하게 된다.
   - 국내 리전을 쓰면 국외 이전 동의 문제는 피하지만, 처리위탁 자체는 남는다.
     개인정보보호법상 위탁 사실 고지·수탁자 관리 의무를 확인한다.
   - 신한라이프의 고객정보 취급 규정상 외부 클라우드 보관이 허용되는지 먼저 확인한다.
     (참고: v1(마누스)에서도 서버 보관 상태였으므로 새로운 위험 범주는 아니다.)
   - 클라우드 선정 시 필수 확인: 저장 리전, 저장 데이터 암호화, 사업자의 데이터 접근 범위,
     **AI 학습 사용 여부**, 계약 종료 시 반출·삭제 절차. 하나라도 불명확하면 채택하지 않는다.
2. **접근 통제 설계** — FC는 자기 고객만, 총관리자는 전체. 팀원 퇴사 시 회수 절차.
3. **백업** — D1 스냅샷 주기, 로컬 내보내기(JSON) 기능 유지.
4. **오프라인 대비** — 상담 현장에서 통신이 끊겨도 고객 조회가 되도록 로컬 캐시 유지.

## 착공 시점 제안

1차 잔여(케어 발행 실전·브리핑 1차)를 마친 뒤 STEP 1부터. 지금 만든 스키마와 화면이
그대로 밑판이 되므로, 중단하고 갈아타는 것보다 얹는 편이 빠르다.
