# 마이가디언 v1 운영 방식 상세 문서

- 용도: v2 개발 시 참조용. 코드 해부 보고서가 아닌 기획 의도 + 실제 운영 실태 중심으로 작성.
- 작성 기준: 코드 직접 탐색 + DB 실데이터 분석 (2026-07-18 기준)
- 추측 표기: 코드에서 확인되지 않은 내용은 [추측]으로 표시

## 1. 고객관리

### 1-1. 고객 등록부터 상담·계약까지의 화면 흐름

고객관리의 진입점은 대시보드 하단 탭바의 고객(CustomersTab) 탭이다. 이 탭은 데스크톱에서는 테이블 뷰, 모바일에서는 카드 리스트로 자동 전환되며, 가상 스크롤(30명 이상 시 @tanstack/react-virtual 활성화)로 대량 고객도 부드럽게 처리한다.

등록 방법은 세 가지로 설계되었다.

첫째, 수동 등록이다. 고객 추가 버튼을 누르면 이름·전화번호·생년월일·성별·주소·카테고리를 입력하는 폼이 열린다. 이 방식이 가장 많이 쓰였으며, DB 기준 270명 중 196명(72.6%)이 manual로 등록되어 있다.

둘째, 이미지 OCR 등록이다. 명함이나 고객 서류 사진을 업로드하면 AI가 텍스트를 추출해 고객 정보를 자동 입력한다. 70명(25.9%)이 이 방식으로 등록되었으며, 이관 고객 일괄 등록에 주로 활용된 것으로 보인다.

셋째, 보장분석 연동 등록이다. 스마트보장분석 리포트 업로드 시 고객이 자동 생성되는 경로로, 4명이 이 방식으로 등록되었다.

등록 후 상담 흐름은 고객 카드 클릭 → CustomerDetail.tsx 진입 → 14개 탭 중 필요한 탭으로 이동하는 구조다. 실제 상담 흐름에서 가장 자주 쓰이는 경로는 다음과 같다.

```
고객 등록 → 보유보험 탭(약관 업로드) → 문서함(스마트보장분석 PDF 업로드)
→ AI 결과입력(외부 AI 분석본 등록) → 보장분석(분석 실행)
→ 문서함(엑셀 생성·다운로드) → 카톡카드(고객 발송)
```

### 1-2. 고객 상태·단계 구분 체계

고객 분류는 카테고리(category), 파이프라인 단계(pipelineStage), 등급(grade) 세 축이 독립적으로 존재한다. 이 세 축이 서로 연동되지 않고 독립적으로 관리된다는 점이 v1의 핵심 설계 결정이다.

카테고리 (4종) — 고객의 성격을 구분하는 정적 레이블이다.

| 카테고리 | 정의 | 실제 건수 |
|---|---|---|
| transfer (이관) | 이전 회사·담당자에서 넘어온 고객. 기존 계약 분석이 주목적 | 70명 |
| valueup (밸류업) | 기존 고객 중 추가 제안 대상. 보장 업그레이드가 주목적 | 172명 |
| contract (계약) | 계약 완료 고객. 유지·관리 단계 | 7명 |
| referral (지인) | 소개·지인 경로로 유입된 고객 | 21명 |

카테고리 간 이동은 가능하도록 설계되었다(예: 이관 → 계약). 이관 고객이 계약을 완료하면 contract로 변경하는 방식이다.

파이프라인 단계 (5단계) — 영업 진행 상태를 추적하는 동적 상태값이다.

| 단계 | 의미 |
|---|---|
| new_contact | 첫 접촉 단계 |
| consulting | 상담 진행 중 |
| proposal_sent | 제안서 발송 완료 |
| contracted | 계약 체결 |
| after_service | 계약 후 관리 |

실제 DB를 보면 270명 전원이 new_contact에 머물러 있다. 이는 파이프라인 단계 업데이트 기능이 구현되었으나 실제로 사용되지 않았음을 의미한다. v2에서는 이 기능을 단순화하거나 자동화할 필요가 있다.

등급 (5단계) — 고객의 가치를 나타내는 등급이다.

| 등급 | 의미 |
|---|---|
| vip | 최우선 관리 고객 |
| gold | 고가치 고객 |
| silver | 중간 가치 |
| bronze | 기본 관리 |
| new | 신규 (기본값) |

자동 산정 로직이 있으나(customerGrade 필드), 실제 DB에서는 270명 전원이 new 등급이다. 수동 조정(gradeOverride)도 사용된 흔적이 없다. 등급 자동화 로직이 작동하지 않았거나, 등급 기준이 명확히 정의되지 않아 방치된 것으로 보인다.

### 1-3. 고객코드(C-2026-XXX) 부여 규칙

코드 탐색 결과, customers 테이블에 customerCode 컬럼이 존재하지 않는다. DB의 기본키는 자동 증가 정수 id이며, 화면에서 표시되는 C-2026-XXX 형식의 코드는 클라이언트 사이드에서 id를 포맷팅하여 표시하는 방식으로 구현되어 있다. 즉, 별도의 코드 생성 로직이 없으며 `C-${연도}-${id.toString().padStart(3, '0')}` 형태로 렌더링된다. [추측: 연도는 고객 등록 연도를 사용할 가능성이 높으나, 코드에서 정확한 포맷 함수를 확인하지 못했음]

## 2. 조인웍 (JointWork)

### 2-1. 생성부터 종료까지의 흐름

조인웍은 팀 내 설계사들이 하나의 고객을 공동으로 관리하는 협업 공간이다. 단순한 파일 공유를 넘어 일정·할일·미팅노트·메모·활동로그까지 포함하는 미니 프로젝트 관리 도구로 설계되었다.

생성 흐름:

1. 대시보드 탭바의 조인웍(JointWorkTab) 탭 진입
2. 새 프로젝트 생성 시 제목(고객명 기반), 연결 고객(customerId), 설명 입력
3. 생성자가 자동으로 첫 번째 멤버로 등록됨 (jwMembers)
4. 팀원 초대: 같은 팀(teamId)에 속한 FC만 초대 가능. 초대 시 jwMembers에 추가되며 역할(role) 텍스트를 자유 입력 가능 (예: "보장분석 담당", "프로젝트 리더")

고객 접근 권한: jwProjects.customerId로 특정 고객과 연결되며, 프로젝트 멤버인 FC는 해당 고객의 데이터에 접근 가능하다. 단, 고객 데이터 자체의 소유권(fcId)은 원래 담당 FC에게 있으며, 조인웍은 협업 뷰를 제공할 뿐 소유권을 이전하지는 않는다.

종료: status를 completed로 변경하는 방식. 실제 DB에서는 7개 프로젝트 모두 active 상태이며, 완료된 프로젝트가 없다.

### 2-2. 조인웍 관련 테이블 8개 역할과 관계

```
jwProjects (7건)
  ├── jwMembers (14건)       ← 프로젝트 참여 FC 목록 (1프로젝트당 평균 2명)
  ├── jwFiles (77건)         ← 공유 파일 (S3 저장, fileKey + fileUrl)
  ├── jwSchedules            ← 공동 일정 (startAt/endAt UTC ms)
  ├── jwTasks                ← 역할 분담 할일 보드 (todo/in_progress/done)
  ├── jwMeetingNotes         ← 미팅 노트 (customerNeeds, keyPoints, nextActions)
  ├── jwMemos                ← 팀 내 커뮤니케이션 메모 (핀 고정 가능)
  ├── jwActivityLogs         ← 모든 활동 자동 기록 (action, targetType, targetId)
  └── jwSharedResources      ← 앱 내 자료 공유 (보장분석/케어콘텐츠/제안서 링크)
```

jwComments와 jwReactions도 스키마에 존재하며, 메모·미팅노트에 댓글과 이모지 리액션을 달 수 있도록 설계되었다.

### 2-3. 수수료·배분 관련 필드

jwProjects, jwMembers 어디에도 수수료(commission), 배분(split), 비율(ratio) 관련 컬럼이 존재하지 않는다. 조인웍은 순수 협업 공간으로만 설계되었으며, 수수료 배분 기능은 v1에서 구현되지 않았다. jwMembers.role은 자유 텍스트 필드로, 역할 명칭만 저장한다.

### 2-4. 실제 등록된 조인웍 77건의 성격

jwFiles 77건은 실사용 데이터다. 7개 프로젝트 모두 실제 고객(customerId)과 연결되어 있으며, 고객명이 프로젝트 제목에 그대로 사용되었다(예: "김창호(남/56/오산)", "추명화(여/46/안양)"). 파일 업로드는 주로 약관 PDF를 조인웍 공간에 공유하는 용도로 사용된 것으로 보인다.

7개 프로젝트의 생성 시기는 2026년 3월 30일 ~ 5월 29일로, 약 2개월간 실제 업무에서 활용되었다.

## 3. 케어센터

### 3-1. 케어 콘텐츠 발행 방식 (주간·월간 안창민)

케어센터는 v1 개발 과정에서 여러 차례 전면 재설계된 기능이다. 최종 형태는 두 가지 콘텐츠 채널로 정착했다.

월간 안창민 (Magazine): AI가 3가지 주제를 제안하면 설계사가 선택 → AI가 칼럼 초안 생성 → 설계사가 인앱 편집(AI 수정 루프 포함) → 발행. 총 4호 발행, 12개 아티클 생성. 커버 이미지는 AI 생성 또는 직접 업로드.

주간 안창민 (Weekly): 매주 발행하는 뉴스레터 형식. 총 11호 발행(2026년 4월 2주차 ~ 7월 2주차). 모두 published 상태이며, 조회수는 호당 2~23회로 실제 고객이 열람했음을 확인할 수 있다. 커버 이미지는 AI 생성 또는 직접 업로드.

발행 후 고객 전달 방식: careDeliveries 테이블에 24건의 발송 기록이 있다. 각 발송 건은 shareToken(고유 URL 토큰), kakaoMessage(카톡 발송 문구), isSent, viewCount, pdfDownloadCount를 추적한다. 고객별 개인화 태그(personalizedTags)도 지원한다. 실제 발송은 카카오톡 메시지 복사 후 수동 전송 방식이다.

케어 토픽 히스토리: careTopicHistory 35건은 AI가 제안한 주제 이력이다. 카테고리는 health(건강), market(시장), tax(세금), coverage(보장), news_economy(경제뉴스) 5종이다.

### 3-2. 케어센터 탭 구조 (최종 형태)

v1 최종 버전의 케어센터는 CareTab.tsx(66줄)로 매우 단순화되어 있다. 이전에는 케어 시리즈, 발송 이력, 콘텐츠 카드 등 복잡한 구조였으나, v8.7 대규모 개편에서 주간/월간 두 채널로 단순화되었다. 대시보드 탭에서 케어 탭을 누르면 주간·월간 탭으로 분기된다.

| 설정 | 알림 목록, FC 프로필 설정, 팀 관리, 총 관리자 | 사용됨 |

총 탭/메뉴 항목 수: 메인 탭 8개 + 더보기 17개 + 고객 상세 14개 = 39개

## 5. 기타 운영 규칙

### 5-1. 알림·리마인더 자동화

pipelineJobs 테이블에 88건의 작업 기록이 있다. 이 테이블은 AI 분석, 엑셀 생성 등 비동기 작업의 상태를 추적한다. 별도의 알림 자동화(예: 생일 알림, 계약 만기 알림)는 스키마에 설계되어 있으나 실제 발송 기록이 없다. 알림 기능은 notifyOwner 헬퍼를 통해 FC 본인에게만 발송되는 구조이며, 고객에게 자동 발송되는 기능은 없다.

### 5-2. AI 분석 기능 입출력 구조

v1에서 AI가 실제로 호출되는 지점은 다음과 같다.

| 기능 | 진입 화면 | 입력 | 출력 |
|---|---|---|---|
| 스마트보장분석 파싱 | 문서함 → PDF 업로드 | PDF 파일 (S3 URL) | 고객 기본정보 + 보유보험 + 7대영역 점수 (JSON) |
| AI 결과입력 | AI 결과입력 탭 | ChatGPT 분석 마크다운 텍스트 붙여넣기 | 파싱된 보험 데이터 → external_ai_results 저장 (32건) |
| 보장분석 실행 | 보장분석 탭 → 새 분석 | 보유보험 데이터 + AI 분석본 | 7단계 분석 결과 JSON + 보장점수 (21건 완료) |
| 엑셀 생성 | 문서함 → 분석리포트 버튼 | 저장된 보장분석 결과 | A3 엑셀 파일 (AI 호출 없음, 포맷 변환만) |
| 카톡카드 생성 | 카톡카드 탭 | 고객 정보 + 보유보험 요약 | 딥네이비+골드 디자인 카드 이미지 (html2canvas) |
| 미팅노트 | 미팅노트 탭 | 음성 녹음 파일 | Whisper 텍스트 변환 → 구조화 노트 |
| 케어 콘텐츠 | 케어 탭 | 주제 선택 + 고객 데이터 | 칼럼/뉴스레터 초안 마크다운 |
| 맞춤제안서 | 맞춤제안서 탭 | 고객 보장 갭 분석 결과 | 제안서 초안 |

핵심 원칙: 엑셀 생성은 AI를 호출하지 않는다. 저장된 분석 데이터를 포맷 변환만 한다. 이 원칙은 v2에서도 유지해야 한다.

### 5-3. 설계 당시 "이건 꼭 지키자"고 정한 원칙

코드 주석, CODEMAP.md, COVERAGE_FLOW.md에서 확인된 절대 원칙들이다.

데이터 무결성 원칙:

- 스마트보장분석 데이터는 그대로 추출만 한다. 변환·합산·매핑 절대 금지.
- 스마트보장분석 리포트는 반드시 PDF. 파일명 패턴: `이름_스마트보장분석리포트.pdf`
- "분석 리포트" 버튼 = 읽기전용 엑셀 뷰어 (id=0, readOnly=true). "수정" 버튼 = 인앱 SpreadsheetEditor (id=doc.id). 두 버튼의 역할을 절대 혼동하지 않는다.

코드 품질 원칙:

- 한번 만든 로직은 더럽히지 않는다. 패치 위에 패치를 쌓지 않는다.
- 라우터 파일은 150줄 이하로 유지. 초과 시 server/routers/<feature>.ts로 분리.

UI/UX 원칙:

- 토스(Toss) 스타일 — 직관적이고 심플한 금융 UI.
- 다크/라이트 테마 지원.
- 인쇄 최적화 (A3/A4 출력 시 가독성 우선).
- 모바일 퍼스트 — 모든 기능은 모바일에서도 동작해야 한다.

워크플로우 원칙 (5단계):

1. 스마트보장분석 PDF 업로드 → 자동분석 → 보유보험 등록
2. 약관/상품설명서 AI 분석(ChatGPT) → 분석본 등록 → 보유보험 매칭 → AI 통합분석
3. 보장분석 새 분석 실행
4. 엑셀 작성 → 분석리포트(읽기전용) → 수정(인앱편집)
5. A3 출력 → 고객 상담

## 6. v2 개발 시 참조 포인트

### 6-1. 버려도 되는 기능 (실사용 0 또는 거의 없음)

- 파이프라인 단계 자동 추적 — 270명 전원이 new_contact. 수동 업데이트가 이루어지지 않았음.
- 고객 등급 자동 산정 — 270명 전원이 new. 등급 기준이 실무에서 정의되지 않았음.
- 태그 탭 — 사용 흔적 없음.
- 계약증서 탭 — 사용 흔적 없음.
- 상세정보 탭 (직업·가족·소득) — 사용 흔적 없음.
- 코칭 기능 — v1에서 이미 제거됨 (v9.1 #91). 외부 링크로 대체.
- 추천/소개 관리 — referralLinks, referralSubmissions 테이블 있으나 사용 흔적 없음.
- 사진 일괄 업로드 — 사용 흔적 없음.
- 위임장 관리 — 사용 흔적 없음.
- 게임 기능 (안봉두 RPG, 크로스 보상함) — 테스트 목적으로 추가, 실사용 없음.

### 6-2. 반드시 살려야 하는 기능 (실사용 확인)

- 보유보험 등록 + 약관 업로드 (465건 policy 문서)
- 스마트보장분석 PDF 파싱 (14건 smart_report)
- AI 분석본 등록 (외부 AI 결과입력) (43건 ai_analysis, 32건 external_ai_results)
- 보장분석 실행 + 엑셀 생성 (21건 완료)
- 카톡카드 생성 (핵심 영업 도구)
- 주간 안창민 (11호 발행, 실제 조회 발생)
- 조인웍 (7개 프로젝트, 77개 파일, 실사용)
- 지식 허브 (세법·보험·멘트 DB)

### 6-3. 개선이 필요한 기능

- 파이프라인 단계 — 자동 업데이트 로직 추가 또는 제거 결정 필요.
- 고객 등급 — 실무 기준(총 보험료, 보유 건수, 최근 상담일)으로 자동 산정 로직 구체화 필요.
- 문서함 파일 다운로드 — CloudFront 한글 파일명 Access Denied 버그 (v1에서 수정 완료, v2에서 동일 구조 사용 시 주의).
- 조인웍 수수료 배분 — 설계사들의 실제 니즈가 있으나 v1에서 미구현. v2 검토 대상.
