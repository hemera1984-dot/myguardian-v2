# 고객관리 분리 설계 — 총관리자 / 팀원별 접근 통제

원칙: 분리는 고객관리에만 적용한다. 공개 플랫폼(이 저장소)에는 로그인이 없다(헌법 유지).
로그인을 직접 만들지 않고 GitHub 권한 모델을 인증으로 쓴다.

## 구조

```
myguardian-v2 (public)          ← 플랫폼. 고객 데이터 절대 금지. 로그인 없음.
mg-clients-<FC코드> (private)   ← FC 1인당 저장소 1개. 고객 실데이터는 여기에만.
  C-2026-001/
    profile.json                ← customer.schema.json 규격 (코드·카테고리·상담일)
    memo.md                     ← 상담 메모
    card/                       ← 카톡카드 산출물
  C-2026-002/
  ...
```

## 다기기 동기화 (2026-07-19 확정)

여러 PC(회사 노트북·집 노트북)에서 같은 고객 자료를 보려면 FC별 비공개 저장소를 쓴다.
원드라이브 대신 git을 쓰는 이유: 충돌을 조용히 덮어쓰지 않고 알려주며, 이력이 남아 복구된다.

### 구조적 전제 — 브라우저는 git을 실행할 수 없다

고객관리 화면은 **로컬 폴더(저장소 복제본)** 를 직접 읽고 쓴다. 화면에서 등록·수정·삭제·이관을
하면 그 순간에는 이 PC의 파일만 바뀐다. GitHub에 반영하려면 동기화 단계가 따로 필요하다.
브라우저에 GitHub 토큰을 넣는 방식은 정적 사이트에 비밀키를 두는 것이므로 금지한다.

### 운영 절차 (PC마다 동일)

1. 최초 1회: `git clone https://github.com/<계정>/mg-clients-<코드>.git C:\projects\mg-clients-<코드>`
2. 작업 시작 전: 바탕화면 "고객자료 동기화" 실행 (다른 PC 변경분 받아옴)
3. 고객관리 화면에서 작업 (폴더 연결 → 등록·메모·이관)
4. 작업 종료 후: "고객자료 동기화" 다시 실행 (이 PC 변경분 올림)

동기화 도구: `myguardian-launcher\고객자료 동기화.bat` — fc01·fc02·fc03을 순회하며
커밋 → pull --rebase → push 한다. 충돌이 나면 덮어쓰지 않고 중단하고 안내한다.

### 팀원 확장

- FC 1인당 비공개 저장소 1개. 팀원은 자기 저장소만 collaborator로 초대한다.
  총관리자(안창민)는 전 저장소 owner이므로 셋 다 clone 가능하다.
- 팀원 PC에도 git이 필요하다. 명령줄이 부담되면 GitHub Desktop(GUI)에서
  Fetch·Push 두 버튼으로 같은 일을 할 수 있다. 동기화 배치 파일을 배포해도 된다.
- 저장소에 GitHub Pages를 절대 켜지 않는다 (개인정보 공개 위험).

### 한 대에서만 쓸 경우

상담을 한 기기에서만 한다면 동기화가 필요 없다. 폴더를 그 PC에 두고 쓰면 되며,
백업 목적으로만 가끔 동기화를 실행한다.

## 권한 = 접근 통제

| 역할 | 권한 |
|---|---|
| 총관리자 (안창민) | 모든 mg-clients-* 저장소의 owner. 전체 열람·관리 |
| 팀원 FC | 자기 mg-clients-<본인코드> 저장소에만 collaborator로 초대됨 |

- "로그인" = GitHub 계정 인증. 비밀번호 관리·인증 서버 개발이 불필요하고 2단계 인증도 GitHub이 제공한다.
- 팀원은 타인의 고객 저장소가 목록에조차 보이지 않는다.

## 사용 방식

- FC는 자기 저장소를 로컬에 클론해서 쓴다. 고객 등록 = 폴더 생성(폴더명이 곧 고객코드).
- 플랫폼 도구(카톡카드 생성기 등)는 브라우저에서 로컬 파일을 불러 쓴다. 외부 전송 없음(헌법 준수).
- GitHub 웹·모바일 앱에서도 자기 저장소 열람 가능.

## 금지 사항

- 고객 저장소에 GitHub Pages를 절대 켜지 않는다. 비공개 저장소여도 Pages 사이트는 공개된다.
- 공개 플랫폼 저장소에 고객 실명·연락처·식별 가능 정보 커밋 금지 (헌법 개인정보 조항).
- 고객 저장소를 public으로 전환 금지.

## 2차 공사 연계

웹 화면에서의 로그인·실시간 동기화가 필요해지면 Cloudflare Workers + Access를 얹는다.
저장소 구조는 그대로 쓰므로 이 설계는 이전 비용이 없다.

## 개설 절차 (팀원 합류 시)

1. 총관리자 계정으로 `mg-clients-<FC코드>` private 저장소 생성
2. 해당 FC의 GitHub 계정을 collaborator로 초대
3. 저장소에 customer.schema.json 사본과 폴더 규칙 README 배치
