# 고객관리 분리 설계 — 총관리자 / 팀원별 접근 통제

원칙: 분리는 고객관리에만 적용한다. 공개 플랫폼(이 저장소)에는 로그인이 없다(헌법 유지).
로그인을 직접 만들지 않고 GitHub 권한 모델을 인증으로 쓴다.

## 구조

```
myguardian-v2 (public)          ← 플랫폼. 고객 데이터 절대 금지. 로그인 없음.
mg-clients-<FC코드> (private)   ← FC 1인당 저장소 1개. 고객 실데이터는 여기에만.
  C-2026-001/
    profile.json                ← customer.schema.json 규격 (코드·카테고리·상담일)
    memo.md                     ← 상담 메모
    card/                       ← 카톡카드 산출물
  C-2026-002/
  ...
```

## 권한 = 접근 통제

| 역할 | 권한 |
|---|---|
| 총관리자 (안창민) | 모든 mg-clients-* 저장소의 owner. 전체 열람·관리 |
| 팀원 FC | 자기 mg-clients-<본인코드> 저장소에만 collaborator로 초대됨 |

- "로그인" = GitHub 계정 인증. 비밀번호 관리·인증 서버 개발이 불필요하고 2단계 인증도 GitHub이 제공한다.
- 팀원은 타인의 고객 저장소가 목록에조차 보이지 않는다.

## 사용 방식

- FC는 자기 저장소를 로컬에 클론해서 쓴다. 고객 등록 = 폴더 생성(폴더명이 곧 고객코드).
- 플랫폼 도구(카톡카드 생성기 등)는 브라우저에서 로컬 파일을 불러 쓴다. 외부 전송 없음(헌법 준수).
- GitHub 웹·모바일 앱에서도 자기 저장소 열람 가능.

## 금지 사항

- 고객 저장소에 GitHub Pages를 절대 켜지 않는다. 비공개 저장소여도 Pages 사이트는 공개된다.
- 공개 플랫폼 저장소에 고객 실명·연락처·식별 가능 정보 커밋 금지 (헌법 개인정보 조항).
- 고객 저장소를 public으로 전환 금지.

## 2차 공사 연계

웹 화면에서의 로그인·실시간 동기화가 필요해지면 Cloudflare Workers + Access를 얹는다.
저장소 구조는 그대로 쓰므로 이 설계는 이전 비용이 없다.

## 개설 절차 (팀원 합류 시)

1. 총관리자 계정으로 `mg-clients-<FC코드>` private 저장소 생성
2. 해당 FC의 GitHub 계정을 collaborator로 초대
3. 저장소에 customer.schema.json 사본과 폴더 규칙 README 배치
