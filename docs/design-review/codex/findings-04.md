## 발견사항

### [치명] 1. 고객 이관이 대상 사본을 검증하지 않고 원본 폴더를 영구 삭제한다

- 위치: `web/customers/index.html:298-311,365-375`
- 재현 방법: `doMove()`의 성공 경로를 따라가면 `copyDir(c.handle, dst)`가 반환된 직후 `c.parent.removeEntry(c.code, { recursive: true })`를 실행한다. 대상 폴더의 상대 경로 목록, 파일 수, 크기, 내용 해시 또는 재열람 검증은 한 번도 수행하지 않는다.
- 근거: 요청서와 화면 설명은 이관 순서를 `복사 → 검증 → 원본 삭제`로 정의하지만 실제 구현은 `복사 → 원본 삭제`다. `createWritable().close()`의 성공만으로 대상 사본 전체가 원본과 동일하다는 애플리케이션 차원의 검증을 대체할 수 없다. 삭제는 복구 수단 없이 재귀 실행된다.
- 수정 제안: 원본 파일 매니페스트를 상대 경로·크기·해시로 만들고, 대상 파일을 다시 열어 전건 일치한 경우에만 원본 삭제 버튼을 활성화한다. 검증 실패 시 원본은 유지하고 대상 불완전 사본의 위치와 정리 실패 여부를 명시한다. 원본 삭제 전 고객코드와 검증 결과를 다시 확인시키는 단계도 필요하다.

### [중요] 2. 이관 대상 검사가 원본의 부모 한 곳만 막아 원본 내부로 자기복사할 수 있다

- 위치: `web/customers/index.html:343-374`
- 재현 방법: 연결된 고객 `C-2026-001`의 고객 폴더 자체를 이관 대상으로 선택하는 모형 핸들을 실제 화면에 주입했다. `target.isSameEntry(c.parent)`는 `false`가 되어 통과했고, 원본 안에 다시 `C-2026-001`을 만든 뒤 그 폴더를 계속 재귀 복사했다. 검수 모형은 깊이 제한 전까지 파일 쓰기 10회를 반복했고, 제한 오류 뒤에야 중단됐다.
- 근거: 현재 검사는 대상과 고객의 직계 부모가 같은지만 확인한다. 대상이 고객 폴더 자체인지, 고객 폴더의 하위인지, 연결 저장소 안의 다른 폴더인지는 막지 않는다. 실제 환경에서는 용량 소진·장시간 멈춤·불완전 대상 폴더를 만들 수 있다. 대상 정리 실패도 무시되므로 이후 같은 코드의 이관이 계속 차단될 수 있다.
- 수정 제안: 대상이 허용된 `mg-clients-*` 저장소 루트인지 확인하고, 원본 고객 핸들·부모·연결 루트 및 원본 아래 모든 디렉터리와 `isSameEntry`를 비교해 동일 트리 선택을 거부한다. 대상 정리에 실패하면 일반 실패 문구가 아니라 불완전 사본 경로와 수동 정리 필요성을 표시한다.

### [중요] 3. 발행 검증기가 자료형이 잘못된 본문을 통과시켜 리더가 깨질 수 있다

- 위치: `pipeline/publish_care_issue.py:27-64`, `web/care/issue.html:74-120`, `data/schema/care-issue.schema.json`
- 재현 방법: 기사 `본문`, `요약`과 `편집장의말`에 비어 있지 않은 문자열을 넣으면 현재 검증은 모두 참으로 보아 통과한다. 발행 후 리더는 각 값에 `.forEach()`를 호출하므로 `TypeError`가 발생한다. 본문 `id`와 메타 `id` 일치, 채널별 기사 수, 기사 번호 중복도 검사하지 않는다.
- 근거: 메타 16건은 개정 스키마를 통과했고 현재 초안은 18개 빈 항목으로 정상 차단됐다. 그러나 본문 구조를 위한 스키마가 없고, 검증기는 존재 여부만 확인한다. 따라서 잘못된 구조가 `상태: 발행`으로 전환되는 경로가 남아 있다.
- 수정 제안: 본문 전용 JSON Schema를 추가해 `편집장의말`, `요약`, `한마디`, `본문`의 배열형과 항목형을 검증한다. `body.id == entry.id`, 주간 4개·월간 3개 기사, 기사 번호의 유일성·연속성, 빈 문자열 문단 금지도 발행 전 조건으로 둔다.

### [중요] 4. Windows 기본 콘솔에서 발행 명령이 상태와 다른 실패 결과를 낸다

- 위치: `pipeline/publish_care_issue.py:68-73,83-92`
- 재현 방법: Windows PowerShell의 기본 `cp949` 출력에서 `python pipeline/publish_care_issue.py --id weekly-12`를 실행했다. `발행 불가 — 18건`의 `—`를 출력하는 순간 `UnicodeEncodeError`가 발생해 검증 목록 대신 스택 추적만 남았다.
- 근거: 정상 발행 경로도 `issues.json`을 먼저 쓴 뒤 92행에서 같은 문자를 포함한 완료 문구를 출력한다. 따라서 실제로는 발행 상태로 바뀌었는데 명령은 오류 코드와 스택 추적을 반환할 수 있다. 재실행하면 이미 초안이 아니라는 별도 오류가 나와 운영자가 상태를 오판하게 된다.
- 수정 제안: 콘솔 문구를 CP949 호환 문자로 제한하거나 프로그램 시작 시 출력 인코딩을 명시적으로 안전하게 설정한다. 더 근본적으로 임시 파일에 쓴 뒤 원자적으로 교체하고, 상태 변경 성공 여부와 출력 성공 여부가 섞이지 않도록 처리한다.

### [중요] 5. 프로필 사진 값이 이스케이프되지 않아 저장형 속성 주입이 실행된다

- 위치: `web/index.html:236-247,263-266,349-353`
- 재현 방법: 격리된 브라우저에서 `mg_profile.사진`을 `bad" onerror="document.body.dataset.profileXss='executed'`로 미리 저장한 뒤 홈을 열었다. 렌더링된 이미지의 `src`는 `bad`로 끊겼고 `onerror`가 실행돼 표식 값이 `executed`로 바뀌었다.
- 근거: 일반 프로필 문자열은 `esc()`를 거치지만 사진 값은 두 곳에서 `src="..."`에 그대로 연결된다. 화면을 통해 첨부한 사진은 512px JPEG data URL로 정규화되지만, 손상·이관·다른 버전에서 남은 localStorage 값은 검증하지 않는다. 잘못된 JSON 자체는 기본 프로필로 정상 복구됐다.
- 수정 제안: 사진 값은 `data:image/jpeg;base64,` 형식과 최대 길이를 엄격히 검사하고, HTML 문자열 조립 대신 생성한 `img` 요소의 `src` 속성으로 설정한다. 허용 형식이 아니면 사진을 폐기하고 기본 프로필로 복구한다. `localStorage.setItem`의 용량 초과도 사용자에게 표시한다.

### [중요] 6. 수치 카운트업이 실제 데이터가 아닌 중간값을 화면에 표시한다

- 위치: `web/index.html:141-155`
- 재현 방법: 데스크톱 홈을 연 뒤 애니메이션 중간인 약 500ms에 확인했다. 실제 값은 사례 383·용어 418·질병 275인데 카드에는 380·415·273이 표시됐다. 700ms가 지나야 실제 값으로 수렴한다.
- 근거: 헌법은 화면 밀도를 실데이터로만 만들고 가짜 수치를 표시하지 않도록 정한다. 현재 모션은 0부터 실제 값까지 존재하지 않는 수치를 연속 표시하며, 자동 스크린샷과 상담 중 첫 화면에도 틀린 숫자가 남는다.
- 수정 제안: 실제 수치는 처음부터 즉시 표시하고 카드의 불투명도·위치 같은 비수치 요소만 전환한다. 수치 자체를 움직이는 장식 모션은 제거한다.

### [중요] 7. NAVER API HUB 전환 뒤 키 없음 안내는 구 개발자센터를 가리킨다

- 위치: `pipeline/collect_news.py:52-60`
- 재현 방법: 키 파일과 환경변수가 없는 상태에서 수집 명령을 실행하면 `https://developers.naver.com`에서 키를 발급받으라고 안내한다. 같은 파일의 상단 설명은 `https://console.ncloud.com`의 NAVER API HUB를 안내하며 서로 모순된다.
- 근거: 실제 호출 주소와 헤더는 NAVER API HUB 규격으로 올바르게 바뀌었지만, 구 개발자센터 키는 새 호출에 사용할 수 없다. 신규 사용자가 오류 안내를 따르면 호환되지 않는 키를 발급받게 된다. 공식 이관 가이드도 NAVER API HUB에서 새 키를 발급하도록 안내한다.
- 수정 제안: 59행을 NAVER Cloud Platform의 NAVER API HUB Application 경로로 교체하고, 상단 설명과 오류 문구를 하나의 상수 또는 공용 함수로 관리한다.

### [사소] 1. 모바일에서도 전체 페이지 배경이 `fixed cover`로 유지된다

- 위치: `web/assets/platform.css:40-44`, 모바일 390px 플랫폼 화면
- 재현 방법: 390×844 홈에서 문서 높이는 3016px이고 `background-attachment` 계산값은 `fixed`였다. 모바일 미디어 쿼리에는 이를 `scroll`로 되돌리는 규칙이 없다.
- 근거: 배경 파일 자체는 약 87KB로 작지만, 긴 업무 화면에서 고정 배경의 확대·재합성은 모바일 스크롤 비용을 높일 수 있다. iOS 설치 안내까지 지원하는 라운드이므로 모바일 전용 보정이 안전하다.
- 수정 제안: 모바일 미디어 쿼리에서 `background-attachment: scroll`을 적용하고, 필요하면 작은 반복 텍스처나 단색 배경으로 대체한다.

## 통과 항목

- 삭제 확인은 고객코드와 이름을 함께 표시하고, 확정 버튼의 코드로 현재 `ALL`에서 고객을 다시 찾아 다른 고객을 삭제하는 오도 경로가 확인되지 않았다.
- 새 호 채번은 임시 저장소에서 김승은 주간 1·2호를 `weekly-kse-01/02`로 생성했고, 다른 발행인이 같은 slug를 쓰면 기존 ID 충돌로 중단됐다. 안창민 다음 호도 `weekly-13`으로 유일했다.
- `data/care/issues.json` 16건은 `care-issue.schema.json`을 전부 통과했다. 초안 `weekly-12`는 서재·홈·최신호·카톡 문구에서 제외되고 발행 데스크에만 1건 표시됐다. 직접 리더 URL은 요청서의 의도대로 열렸다.
- `care.js`로 옮긴 `kakaoText()` 본문은 이전 구현과 공백 한 줄을 제외하고 동일했다. 주간 11호 카톡 문구의 제호·4개 꼭지·서명도 기존 양식으로 생성됐다.
- 프로필의 깨진 JSON은 기본 프로필로 복구됐다. 정상 사진은 512px JPEG로 축소된다.
- 카톡카드의 단색·골드 프레임·축하 배경을 실제 `html2canvas`로 렌더링했다. PNG 크기와 SHA-256이 각각 달라 두 로컬 배경 이미지가 결과물에 포함됨을 확인했다.
- 이미지 참조 경로는 모두 존재했다. `hero-team`, `hero-care-2`, `hero-report-2`만 요청서에 명시된 예비 자산으로 미사용 상태였다.
- 케어 사이드바는 390px에서 메뉴 열기, 오버레이, `aria-expanded`가 함께 전환됐고 가로 스크롤은 없었다. 1180·1920px 주요 화면에서도 가로 초과와 콘솔 오류가 없었다.
- NAVER API HUB 호출 도메인·경로·인증 헤더는 공식 이관 규격과 일치했다.
