## 검수 방법

- `data/cases/cases.json` 726건을 내용 기준으로 별도 집계했다. 사건번호·원문 URL·복합키 중복, 등급과 원문 존재 여부, 요약 품질, 출처별 번호 규칙을 확인했다.
- 서로 다른 원문 URL 40개를 고정 표본으로 요청했다. 40개 모두 HTTP 200이었으나, 일부 법원 링크는 다른 법령 페이지로 이동해 사건번호·제목 일치 여부를 추가 확인해야 했다.
- `scripts/screenshot.py`로 홈·청구·사례·학습·고객·케어 서재·케어 리더·카톡카드·리포트·운영실의 1920/1180/390px 화면 30장을 다시 생성했다. 세 해상도에서 명백한 가로 스크롤과 깨진 이미지는 없었다.
- 고객관리는 개인정보가 없는 메모리 픽스처로 직접 연결했다. 루트 고객 1명과 FC 하위 폴더 고객 1명을 구성해 목록·검색·상세를 확인했다.
- 자가 검증 문서에 이미 적힌 단순 검색, `div` 행 키보드 접근, 사이드바·이스케이프 함수 중복 등은 재발견 항목으로 반복하지 않았다. 아래에는 추가 결함 또는 자가 검증보다 범위가 큰 결함만 기록했다.

## 발견 사항

### [치명] 사례 726건 중 343건이 동일 사건·동일 원문 중복이며 홈 통계도 중복을 실데이터로 표시한다

- 수정 제안: 원본 보존층과 서비스용 색인을 분리한다. 원본은 변형하지 말고, 서비스 색인 생성 단계에서 공식 원문 식별자 우선의 중복 그룹을 만든 뒤 대표 레코드와 원본 ID 목록을 함께 보존한다. 통계와 검색 건수는 서비스 색인 기준으로 다시 생성하고 중복 판정 보고서를 남긴다.

### [치명] 금감원 129건 중 82건의 표시 사건번호가 공식 URL 식별자와 불일치하거나 원문이 없다

- 위치: `data/cases/cases.json:10587`, `data/cases/cases.json:10595` 등 금감원 레코드; 사례 상세의 사건번호·검증완료 배지
- 재현 방법: `금감원-분쟁조정-N`의 끝 숫자와 `case_source_url`의 `caseSlno`를 비교한다. 129건 중 82건이 불일치 또는 원문 없음이다. 예를 들어 화면 사건번호 `금감원-분쟁조정-100`의 링크는 `caseSlno=101`이다. 이 값은 공식 조정번호가 아니라 이관 순번으로 보인다.
- 근거: 스키마는 `case_number`를 “사건번호 또는 조정번호”라고 정의하고(`data/schema/case.schema.json:14`), 헌법은 사건번호 없는 판례를 표시하지 못하게 한다. 검증완료 배지는 식별자가 공식 출처와 대응할 때만 정당하다.
- 수정 제안: 금감원 원문에서 공식 사건·조정 식별자를 추출해 별도 검증한다. 공식 번호가 없는 공개 분쟁사례는 `case_number`에 임의 번호를 만들지 말고 자료 식별자 필드를 별도로 두며, 스키마와 표시 규칙을 분쟁사례/판례에 맞게 분리한다. 검증 전에는 등급을 `부분검증` 이하로 낮춘다.

### [치명] “검증완료” 726건의 내용 품질이 등급 의미를 충족하지 않는다

- 수정 제안: 검증완료의 체크리스트를 `공식 원문 또는 신뢰 가능한 원문`, `식별자 일치`, `제목·요약 대조`, `중복 판정`으로 명문화한다. 배치 검증 결과를 레코드별로 저장하고, 하나라도 미충족하면 `부분검증` 또는 `미검증-참고용`으로 재분류한다.

- 근거: 고객 화면은 카카오톡 링크 유입이므로 잘못되거나 누락된 URL에서도 원인을 명확히 알리고 서재로 복구시켜야 한다. 잡지급 가독성 원칙에도 미달한다.
- 수정 제안: 별도 오류 레이아웃을 만들고 “발행물을 찾을 수 없습니다”, “케어센터 서재로 이동” 버튼을 첫 화면 중앙에 충분한 대비로 표시한다. 스크린샷 스크립트에는 실제 발행물 ID가 있는 리더 URL과 오류 URL을 각각 검증하도록 매개변수 지원을 추가한다.

### [중요] 팀원 발행 호에서 문서 제목과 카톡 편집 제목이 발행인을 안창민으로 고정한다

- 위치: `web/care/issue.html:193`, `web/care/index.html:131`
- 재현 방법: `발행인`이 안창민이 아닌 픽스처/향후 호를 넣어도 브라우저 제목과 카톡 문구 편집 패널 제목은 `채널 안창민 N호`로 표시된다. 표지 본문은 `발행 OOO FC · 편집장 안창민 FC`를 올바르게 분리하므로 화면끼리 모순된다.
- 근거: AGENTS.md의 “편집장은 항상 안창민, 팀원 호는 발행 OOO FC 병기” 원칙.
- 수정 제안: 문서 제목과 편집 패널 제목은 실제 `발행인`을 사용하고, 편집장 표기는 별도 메타데이터로 유지한다. 안창민 발행/팀원 발행 두 픽스처로 회귀 검증한다.

### [중요] 케어 메타데이터 스키마와 실제 리더 계약이 서로 다르다

- 위치: `data/schema/care-issue.schema.json:5`, `data/schema/care-issue.schema.json:42-44`; 실제 경로 `data/care/issues.json:37` 이하; `web/care/issue.html`의 JSON 파싱 흐름
- 재현 방법: 스키마 설명은 본문을 `<id>.html` 조각으로 둔다고 명시하지만 15개 호 모두 `<id>.json`을 가리키며 리더도 JSON 객체를 전제로 렌더링한다. `본문파일`이 단순 문자열이라 jsonschema 검사는 이 계약 불일치를 잡지 못했다.
- 근거: 자가 검증은 스키마 통과를 데이터 무결성 근거로 제시했지만, 스키마가 실제 형식을 설명하지 못하면 검증 자체가 무효화된다.
- 수정 제안: 실제 상세 JSON 구조를 별도 스키마로 정의하고 `본문파일` 패턴을 `.json`으로 제한한다. 메타데이터에서 참조한 파일의 존재와 상세 스키마까지 순회 검증한다.

### [중요] 사례 화면은 516KB JSON과 726개 행을 한 번에 DOM에 올린다

- 위치: `data/cases/cases.json` 516,691바이트; `web/cases/index.html:92-110`
- 재현 방법: 최초 진입 시 `list.map(...).join("")`으로 필터 전 726개 행 전체를 생성한다. 모바일 스크린샷에서는 일부만 보이지만 나머지 행도 모두 DOM에 존재한다. 중복 레코드까지 포함해 초기 파싱·문자열 생성·레이아웃 비용을 지불한다.
- 근거: 상담용 태블릿 우선 및 실무 밀도 원칙. 현재 규모에서도 불필요하며 데이터가 늘면 입력 지연과 스크린리더 탐색 비용이 선형 증가한다.
- 수정 제안: 먼저 중복을 제거한 서비스 색인을 만들고, 페이지네이션 또는 점진 렌더링을 적용한다. 최초 50건과 상세에 필요한 필드만 싣고 검색 색인은 별도 경량 구조로 분리한다.

### [중요] 케어 서재 첫 화면이 15개 표지를 전부 내려받아 약 1.70MB를 사용한다

- 위치: `web/care/index.html:164-180`; `data/care/issues/*/cover.jpg`
- 재현 방법: 서재는 필터와 관계없이 15개 카드를 한 번에 만들며 표지 합계와 메타데이터가 약 1,704,500바이트다. 390px 화면에서 사용자는 첫 두 장만 보지만 전부 요청된다.
- 근거: 케어센터는 카카오톡 모바일 유입이 우선이며 네트워크가 불안정한 환경에서도 빠르게 열려야 한다.
- 수정 제안: 표지에 `loading="lazy"`, 폭·높이, 적절한 모바일 파생 이미지를 지정한다. 첫 화면 후보만 우선 로드하고 월간/주간 필터 적용 뒤 필요한 표지만 생성한다.

### [중요] 낮은 대비의 12~13px 보조 텍스트가 플랫폼 전반에 반복된다

- 위치: `web/assets/platform.css:6`, 사용 예시 `web/assets/platform.css:176-177`, `web/assets/platform.css:213-214`, `web/assets/platform.css:344-345`, `web/assets/platform.css:628-629`
- 재현 방법: `#98a2b3`를 흰색 또는 `#fafbfc` 위에 12~13px로 표시한다. 일반 텍스트에 필요한 대비를 충족하지 못하는 구간이 있고, 390px 사례 메타·면책문구·상태 설명에서 특히 작고 흐리다.
- 근거: 접근성 대비 기준과 AGENTS.md의 본문/보조 라벨 위계. “연회색”은 읽을 수 있는 최소 대비를 면제하지 않는다.
- 수정 제안: 보조 텍스트도 4.5:1 이상이 되도록 색을 진하게 하고, 핵심 메타·면책문구는 14px 이상으로 올린다. CSS 변수별 자동 대비 검사와 실제 배경 조합 검사를 추가한다.

### [중요] 케어 표지에 금지된 그라데이션과 영문 장식 카테고리가 남아 있다

- 위치: `web/assets/style.css:1231`; `data/care/issues.json:442`, `data/care/issues.json:448`, `data/care/issues.json:454` 및 월간 상세 JSON
- 재현 방법: 월간 호를 열면 `HOT ISSUE`, `LIFESTYLE`, `INSIGHT`가 장식 라벨로 표시되고 표지에는 선형 그라데이션 오버레이가 적용된다.
- 근거: AGENTS.md의 영문 장식·그라데이션 금지. 케어센터의 세리프·골드 예외는 명시돼 있지만 영문 장식과 그라데이션 예외는 없다.
- 수정 제안: 카테고리를 `주요 이슈`, `생활`, `보험 통찰` 등 한글 기능 라벨로 바꾸고, 표지 가독성은 단색 반투명 면 또는 이미지 자체의 명암 처리로 확보한다. 디자인 예외가 꼭 필요하면 코드보다 먼저 헌법 결정으로 남긴다.

### [제안] CSS에 동일 선택자 재정의가 누적돼 변경 영향 범위를 예측하기 어렵다

- 위치: `web/assets/platform.css` 전체, 특히 파일 후반의 재정의 블록; `web/assets/style.css` 전체
- 재현 방법: 단순 선택자 집계에서 `platform.css`는 중복 선택자 27개이며 `.platform-main` 4회, `.platform-shell`·`.platform-topbar` 3회씩 정의된다. `style.css`도 중복 선택자 8개가 있다. 미디어쿼리 맥락 차이를 감안해도 동일 컴포넌트의 기본 규칙이 여러 후반 블록에서 덮인다.
- 근거: “한번 만든 로직에 패치 위에 패치를 쌓지 않는다”는 v1 계승 원칙. 현재는 어느 선언이 최종 적용되는지 추적 비용이 커진다.
- 수정 제안: 화면 시공과 분리된 리팩터링 작업에서 토큰·기본 레이아웃·컴포넌트·반응형 순으로 CSS를 재구성하고, 같은 맥락의 선택자는 한 곳으로 합친다. 시각 회귀 스크린샷을 기준으로 결과 동등성을 검증한다.

## 총평

**현재 상태는 배포 불가다.** 3해상도 기본 레이아웃은 안정적이고 이미지 파손·가로 스크롤도 발견되지 않았지만, 데이터 신뢰성과 고객 폴더 중복은 화면 완성도보다 우선하는 배포 차단 사유다. 특히 사례 726건이라는 핵심 자산의 약 47%가 동일 원문 기준 초과 중복이고, 금감원 사건번호·검증완료 등급이 실제 검증 상태를 나타내지 못한다. 공개 저장소의 고객 식별정보와 출처 없는 고객 발송 콘텐츠도 즉시 조치가 필요하다.

배포 재심사는 최소한 다음 조건을 충족한 뒤 진행해야 한다.

1. 공개 저장소 고객 식별정보 제거와 이력 노출 점검
2. 사례 중복·사건번호·검증등급 정비 및 통계 재생성
3. 고객 폴더 재귀 중복 수정과 회귀 픽스처 통과
4. 케어 콘텐츠 출처·검증 게이트 도입 및 기존 15개 호 재검증
5. 필터 누락·상세 상태 불일치·케어 오류 진입 상태 수정

그 이후 접근성·초기 로드 비용·CSS 구조를 정리하면 제한된 내부 시험 배포를 검토할 수 있다.
