# 5차 재심 대응 — 3건 수정 완료, 재재심 요청

## 조치 내역

### [치명 1] 검증-확정 사이 변경 허용 → 확정 직전 전건 재검증
- finishMove()가 원본 삭제 전에 **원본 매니페스트를 새로 만들어**(검증 시점이 아니라 확정 시점
  기준) 대상 사본과 경로·크기·SHA-256 전건 재대조한다. 불일치 시 원본을 삭제하지 않고
  사본을 정리한 뒤 사유(어느 파일이 달라졌는지)를 표시한다.
- 원본에 새 파일이 추가된 경우도 새 매니페스트에 포함되므로 대상 누락으로 감지된다.
- 재검증: 대상 파일 변조 후 확정 → 원본 보존·중단, 원본에 파일 추가 후 확정 → 중단. 각 통과.
- 위치: web/customers/index.html finishMove()

### [중요 2] 깊이 제한 우회 → 제한 제거, 전체 순회
- containsHandle()의 depth 인자·제한을 제거했다. 연결 저장소 트리 전체를 끝까지 순회한다.
- 재검증: 연결 저장소 아래 9단계 폴더를 대상으로 지정 → 차단 확인.

### [중요 3] 스키마 미적용 → jsonschema 실검증 적용
- publish_care_issue.py가 care-body.schema.json을 **jsonschema(Draft 2020-12)로 실검증**한다.
  수동 검증과 스키마의 불일치가 원천적으로 불가능해졌다 (검증 주체가 스키마 하나).
- 스키마가 못 보는 정합(본문-메타 id 일치, 채널별 기사 수, 번호 연속·유일, 이미지·커버 실존)만
  코드로 보태며, 자료형이 보장된 뒤에만 실행돼 예외로 죽지 않는다.
- 재검증: Codex 재현 케이스 전건 — 번호 true, 카테고리 누락, 카테고리 숫자, 한마디 0,
  id 패턴 위반, 기사 항목 문자열, 번호 문자열 혼합, 본문 문자열, 편집장의말 문자열 —
  모두 예외 없이 "발행 불가" 목록으로 차단. 유효 자료는 발행 성공.

## 재검증 스위트 저장소 반입 (5차 회귀 지적 대응)
- scripts/verify_review.py — 위 재현을 포함한 자동 검증 27건. 실행:
  `python scripts/verify_review.py` (playwright·jsonschema 필요)
- 발행기 테스트는 임시 사본에서 실행되어 실데이터를 건드리지 않는다.
- 이번 실행 결과: 27건 전건 통과 (이관 차단 3종·변조 중단 2종·정상 이관·취소는 4차분과 합산,
  발행기 유효 1·무효 12, 주입·수치·모바일 배경, 페이지 오류 0).

## 재재심 요청 범위
- finishMove 재검증의 우회 가능성 (재검증과 removeEntry 사이의 잔여 틈은 FSA 특성상
  존재하나, 그 창은 밀리초 단위이고 재검증 실패 시 삭제가 실행되지 않음 — 평가 요청)
- jsonschema 적용 후 스키마 자체의 허점 (경계값 추가 발견 환영)
- scripts/verify_review.py 재실행으로 동일 결과 재현 확인
