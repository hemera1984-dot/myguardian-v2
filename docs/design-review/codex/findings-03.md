## 발견사항

### [치명] 1. 미팅 차수가 이스케이프 없이 `innerHTML`에 들어가 고객 폴더만 열어도 스크립트가 실행된다

- 위치: `web/customers/index.html:154-168`, `web/customers/index.html:265-278`, `web/customers/index.html:533-544`
- 재현: 연결할 고객 폴더의 `profile.json`에 `"미팅":[{"차수":"<img src=x onerror=alert(document.domain)>","일자":"2026-07-21","상태":"예정"}]`을 넣고 폴더를 연결한다. 화면은 프로필을 스키마 검증 없이 읽은 뒤 목록의 `up["차수"]`와 상세의 `m["차수"]`를 그대로 HTML 문자열에 결합한다. 목록 렌더 시점부터 이벤트 속성이 실행될 수 있다.
- 근거: `customer.schema.json`은 차수를 1 이상의 정수로 제한하지만 런타임은 스키마를 적용하지 않는다. 다른 미팅 필드는 `esc()`를 사용하면서 차수만 세 경로에서 빠졌다. 이 앱에는 스크립트 실행을 보조 차단할 CSP도 없다. 비공개 저장소·이관 파일·수동 복구 파일 중 하나가 변조되면 같은 출처에서 임의 JavaScript가 실행돼 현재 화면의 고객 개인정보를 읽을 수 있다.
- 수정 제안: 목록·배지·`aria-label`의 차수를 모두 `esc(String(...))`로 처리한다. 폴더 적재 시 프로필을 스키마 검증하고, 실패한 프로필은 렌더하지 말고 고객코드와 오류만 표시한다. 방어 심층화로 정적 호스팅에 맞는 CSP도 추가한다.

### [중요] 1. 고객 프로필 쓰기가 고객 단위로 직렬화되지 않아 빠른 연속 조작에서 파일과 메모리가 갈라질 수 있다

- 위치: `web/customers/index.html:569-612`, `web/customers/index.html:650-683`
- 재현: 한 고객에서 서로 다른 진행상태 버튼 두 개를 빠르게 누르거나, 미팅 완료·취소·일자 변경을 쓰기 완료 전에 연속 실행한다. `saveStatus()`는 누른 버튼 하나만 잠그고 다른 상태 버튼은 살아 있다. `mutateMeetings()`는 실행 잠금이 전혀 없다. 따라서 같은 `profile.json`에 복수 `createWritable()`이 동시에 열리고 닫히는 순서에 따라 이전 직렬화본이 나중 상태를 덮을 수 있다.

### [중요] 2. 서비스워커의 백그라운드 갱신이 수명에 묶이지 않고 오류 응답도 정상 캐시를 덮는다

- 위치: `web/sw.js:41-68`
- 재현 1: 기존 CSS가 캐시에 든 상태에서 새 HTML과 새 CSS를 배포하고 이동한다. HTML은 네트워크 우선으로 새 버전을 받지만 CSS는 `hit || refresh`에서 이전 캐시를 즉시 반환한다. 캐시가 맞아 `hit`을 반환한 경로에서는 `refresh`를 `event.waitUntil()`로 연장하지 않고, `cache.put()` Promise도 반환하지 않는다. 브라우저가 작업자를 종료하면 갱신이 완료된다는 보장이 없다.
- 재현 2: JSON 또는 이동 요청이 일시적으로 404·500을 반환하게 한다. `fetch()`는 HTTP 오류에서 reject되지 않으므로 기존 캐시 폴백을 쓰지 않고 오류 응답을 그대로 반환하며, 그 응답을 기존의 정상 사본 위에 `cache.put()`한다.
- 근거: v2는 영구적인 캐시 우선보다 개선됐지만, 새 HTML+옛 CSS 혼합을 최소 한 번 허용하고 백그라운드 쓰기의 완료도 보장하지 않는다. 데이터·화면 캐시는 `resp.ok` 검사도 없다. 오프라인 폴백과 업데이트 재발 방지가 확정됐다고 보기 어렵다.
- 수정 제안: 버전 해시가 붙은 자산명 또는 CSS·JS 네트워크 우선 전략으로 HTML과 자산 버전을 결합한다. stale-while-revalidate를 유지한다면 갱신 Promise를 `event.waitUntil()`에 연결하고 `cache.put()` 완료를 await한다. 이동·JSON은 `resp.ok`일 때만 반환·캐시하고, 비정상 HTTP 응답은 기존 정상 캐시로 폴백한다.

### [제안] JSON에서 만든 외부 링크는 HTML 이스케이프와 별도로 URL 스킴을 제한한다

- 위치: `web/claims/index.html:153,218-260`, `web/care/desk.html:310-317`
- 내용: 현재 저장 데이터의 링크는 확인한 범위에서 HTTP(S)지만, `esc()`는 따옴표 삽입만 막고 `javascript:`·`data:` 같은 스킴을 제한하지 않는다. 향후 수집 파이프라인·수동 JSON에서 링크가 들어오는 경로에는 `new URL()`로 파싱해 `https:`와 필요한 `http:`만 허용하는 공용 `safeUrl()`을 둔다.

## 총평과 배포 판정

디자인 개편의 기본 셸, 383건 사례 신뢰도, 고정 고지, 케어 공유 발행의 내용 무변형, 관리자 미노출·정상 트리 순환 차단, 세 해상도 가로 레이아웃은 통과했다. 서비스워커도 고객 FSA 파일을 캐시하지 않으며, 감시 상태 파일에서 개인정보는 발견되지 않았다.

그러나 고객 폴더 데이터로 실행되는 저장형 XSS가 있고, 고객 프로필의 동시 쓰기 무결성과 보험사 기준선 보존이 보장되지 않는다. 서비스워커 갱신도 이전 캐시 고정 문제의 재발을 완전히 차단하지 못했다. 이 세 항목은 고객 실사용·자동 감시·설치형 배포 전에 해소해야 한다.

**배포 판정: 불가**

치명 1건과 중요 6건을 수정한 뒤, 악성·스키마 불일치 프로필, 지연·실패·역순 고객 쓰기, 서비스워커 구버전→신버전 전환·HTTP 오류·오프라인, 보험사 전체·단건·오타 실행, 모바일 탭 순서·44px을 포함해 재검수해야 한다.
