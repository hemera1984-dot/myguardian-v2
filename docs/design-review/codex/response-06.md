# 6차 재심 대응 — 3건 수정 완료, 재재심 요청

## 조치 내역

### [치명 1] 재검증 도중의 원본 변경 → 자동 원본 삭제 자체를 제거 (권고안 채택)
- FSA에 원자적 잠금이 없는 이상 "검증 후 삭제"는 어떤 구현이든 경합 창이 남는다는 결론을
  수용한다. 검증-삭제 연결을 고치는 대신 **연결 자체를 끊었다**:
  - 이관 = 복사 + 매니페스트 전건 검증까지. 완료 문구가 "원본은 보존되어 있습니다"를 명시한다.
  - 원본 정리 = 별도의 삭제 절차(기존 고객 삭제 — 고객코드 명시 2단계 확인)로만 가능하다.
    운영자가 인수 확인 후, 다른 작성자가 없는 시점을 골라 실행한다.
- 자동 삭제 코드(finishMove/cancelMove/pendingMove)는 삭제됐다. 검증과 시간적으로 결합된
  removeEntry 호출이 코드에 존재하지 않으므로 지적된 유실 시나리오가 성립할 수 없다.
- 재검증: 이관 후 원본 보존·사본 내용 일치·"사본 검증 완료/원본은 보존" 문구,
  자동 삭제 UI(#cust-move-yes) 부재, 별도 삭제 절차로 정리(사본 무관) 각 통과.

### [중요 2] 공백 콘텐츠·weekly-00 → 스키마 패턴 강화
- 사람이 읽는 모든 문자열 필드(편집장의말·카테고리·제목·부제·요약·한마디·본문 x)에
  비공백 문자 필수 `pattern: \S` 추가.
- id 패턴을 `(0[1-9]|[1-9][0-9])`로 바꿔 00호를 차단 — care-body와 care-issue 두 스키마 동일 적용.
- 재검증: 공백만으로 구성한 4개 기사 자료 7건 위반 검출·차단, 본문 id weekly-00 차단.

### [중요 3] 최상위 자료형 예외 → 객체 확인 후 접근
- 스키마 검증 후 의미 검사는 `isinstance(body, dict)`일 때만 수행. 아니면 "본문 최상위가
  객체가 아닙니다"를 오류 목록에 더한다. body.get 무조건 호출 제거.
- 재검증: 최상위 배열 본문 → 예외 없이 "발행 불가" 목록으로 종료.
