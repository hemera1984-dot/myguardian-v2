## 발견사항

### [치명] 1. `검증완료` 9건이 어떤 검증 보고서도 거치지 않았다

- 위치: `data/cases/index.json`의 `tribunal_*` 9건 / `pipeline/verify_scourt.py:38` / `pipeline/verify_fss.py:41` / `pipeline/build_case_index.py:54`
- 재현 방법: 색인의 `verification_grade == "검증완료"` ID 집합에서 `fss-verification.json`과 `scourt-verification.json`의 결과 ID 합집합을 뺀다. 조세심판원 9건이 남는다. 이 레코드들은 `source_type`이 빈 문자열이고 CaseNote URL만 있으며 `검증메모`가 없다. 예: `tribunal_조심2014서0940`, `tribunal_조심2014서0941`, `tribunal_조심2014서2441`, `tribunal_조심2013서4496`.
- 근거: 법원 검증기는 `source_type == "대법원"`, 금감원 검증기는 `source_type == "금감원"`만 대상으로 삼는다. 색인 생성기는 보고서에 없는 기존 등급을 유지하고 URL이 없을 때만 강등한다. 따라서 `auto-loop-report.md`의 “전량 원문 실검증”과 최종 `검증완료 218건` 중 9건은 근거가 없다. 헌법의 신뢰도 등급 필수 원칙은 등급 표기뿐 아니라 등급의 근거가 정직해야 한다.
- 수정 제안: 조세심판원을 명시적인 출처 유형으로 정규화하고 별도 공식 원문 검증 보고서에 포함한다. 완료 전에는 9건을 `미검증-참고용` 또는 근거에 맞는 등급으로 내린다. 색인 생성 시 모든 `검증완료`가 검증 보고서 ID 합집합에 포함되는지 실패 조건으로 둔다.

### [치명] 2. 신규 고객 등록이 프로젝트의 고객 스키마와 다른 `profile.json`을 생성한다

- 위치: `web/customers/index.html:358-368` / `data/schema/customer.schema.json:5-21` / `docs/customer-separation.md:8-16`
- 재현 방법: 신규 등록 코드가 쓰는 객체를 `customer.schema.json`에 대입한다. 생성 객체는 `고객코드`, `등록일`, `customerName`, `category`, `phone`이고, 스키마 필수 키 `코드`, `담당FC`, `카테고리`가 모두 없다. 카테고리 값도 스키마의 한글 열거형이 아니라 영문 값이다. 스키마 설명에서 금지한 실명·연락처를 `customerName`, `phone`으로 저장한다.
- 근거: 분리 설계 문서는 `profile.json`을 `customer.schema.json` 규격이라고 명시한다. 현재 생성 파일은 규격 검증에 실패하며, 마이그레이션 파일의 `v1원본` 구조와도 달라 세 번째 프로필 형식을 추가한다. 이는 신규 쓰기 기능의 파일 무결성 결함이다.
- 수정 제안: 먼저 실제 운영 계약을 하나로 확정한다. 개인정보를 private 고객 저장소에 허용할 의도라면 공개 저장소 금지와 private 저장소 허용을 분리해 스키마·문서를 함께 개정하고, UI가 그 단일 스키마만 읽고 쓰게 한다. 현재 스키마를 유지할 의도라면 신규 등록은 필수 키와 한글 열거형만 기록하고 이름·연락처 입력을 제거한다. 어느 쪽이든 저장 직전 스키마 검증과 폴더명-코드 일치 검증을 추가한다.

### [중요] 1. 최장 요약 대표 선정이 35개 중복 그룹의 쟁점분류를 임의로 버린다

- 위치: `pipeline/build_case_index.py:27-44`, 사례 화면의 쟁점 필터
- 재현 방법: 원본을 `(case_number, case_source_url)`로 그룹화한 뒤 그룹 안의 고유 `issue_category`가 둘 이상인 그룹을 센다. 35개다. 대표 예시는 `서울행정법원-2015-구합-58560`의 `보험료납입 / 증여세·보험`, `93누1664`의 `보험금청구 / 보험금지급`, `2009다59688`의 `보험금청구 / 보험계약해지`다. 이 그룹들은 요약 길이도 각각 모두 같아 `max()`가 첫 입력 레코드를 고른다.
- 근거: 중복 건수와 통계 왜곡은 해소됐지만, 대표 선정 기준이 동률일 때 입력 순서에 의존한다. 서비스 색인이 한 쟁점만 남기므로 다른 쟁점으로 필터링하면 같은 사건을 찾지 못한다. “데이터는 변환·합산·매핑하지 않는다”는 헌법 아래에서 의미 필드를 조용히 소실시키는 선택은 추적 가능해야 한다.
- 수정 제안: 대표 레코드 선택과 별도로 그룹의 원본 쟁점분류 목록을 추적 필드로 보존하고 필터는 그 목록을 사용한다. 또는 동률·분류 상충 그룹을 수동 판정 큐로 보내 하나의 공식 분류를 정한다. 자동 병합으로 새 의미를 만들지는 않는다.

### [중요] 2. 카톡카드 월 보험료 합계가 정상적인 금액 표기를 잘못 계산한다

- 위치: `web/card/index.html:89-116`
- 재현 방법: 보유보험에 `상품A | 12,000원`, `상품B | -45,000원`, `상품C | 1.2만원`을 입력한다. 화면 합계는 `57,012원`이 된다. 숫자가 아닌 모든 문자를 제거해 음수 부호를 없애고 `1.2만원`을 `12`로 해석하기 때문이다.
- 근거: 고객에게 발송하는 카드에 실제와 다른 금융 수치가 표시된다. 헌법의 데이터 무변형·합산 금지 원칙과도 충돌한다. 특수문자 XSS 방어는 통과했지만 값의 의미 검증은 없다.
- 수정 제안: 합계를 제거하고 입력 원문만 표시하거나, 허용 입력을 원 단위 비음수 정수로 제한해 정규식 전체 일치·범위 검증을 통과한 값만 계산한다. `만원`, 소수, 음수 등 허용하지 않는 형식은 합계에서 조용히 변환하지 말고 해당 행에 오류를 표시한다.

### [중요] 3. 고객 쓰기 작업은 일부 저장과 중복 실행을 막지 못한다

- 위치: `web/customers/index.html:297-315`, `web/customers/index.html:340-374`
- 재현 방법: 신규 등록은 폴더와 `profile.json`을 먼저 만든 뒤 `memo.md`를 만든다. 마지막 쓰기가 실패하면 오류 문구를 내지만 앞서 만든 폴더와 프로필은 남는다. 메모 저장 버튼은 신규 등록 버튼과 달리 실행 중 비활성화하지 않아 빠른 연속 클릭 시 같은 파일에 복수 `createWritable()`이 열린다.
- 근거: 요청한 “파일 무결성”과 “더블클릭 가드”가 신규 등록 버튼의 첫 진입에만 적용됐다. 실패 후 재시도하면 부분 생성 폴더가 코드 충돌 회피 대상이 되어 빈 번호와 불완전 고객이 남을 수 있다.
- 수정 제안: 신규 등록은 임시 폴더 또는 완료 표식을 사용해 모든 파일 저장 성공 뒤에만 유효 고객으로 인식한다. 실패 시 생성물을 정리하거나 복구 가능한 상태를 명시한다. 메모 저장에도 실행 중 잠금과 완료 후 해제를 적용하고 동일 고객별 저장을 직렬화한다.

### [제안] 1. 법원 검증 산출물만으로는 나중에 판정을 재현하기 어렵다

- 위치: `pipeline/verify_scourt.py:55-95`, `data/cases/scourt-verification.json`
- 재현 방법: 법원 결과 레코드는 `id`, 표시번호, 판정, 사유만 저장한다. 실제 요청 URL·최종 리디렉션 URL·응답 상태·확인 시각·제목 대조 결과가 없다. 검증 로직도 렌더링된 전체 `body`에 정규화 사건번호가 포함되는지만 본다.
- 근거: 이번 표본은 보수적으로 판정됐지만, 페이지 개편이나 리디렉션 변경 뒤 어떤 화면을 확인했는지 결과 파일만으로 감사할 수 없다. 1차 대응 문서가 제시한 체크리스트의 “원문 접근·식별자 일치·제목/요약 대조” 중 제목·요약 대조도 자동 공정에 없다.
- 수정 제안: URL, 최종 URL, HTTP/탐색 결과, 검증 시각, 정규화 사건번호, 제목 대조 결과를 기록한다. `검증완료`는 사건번호뿐 아니라 제목 또는 핵심 요약 대조까지 통과한 경우로 명문화한다.

### [제안] 2. 공용 스타일 끝에 보정 규칙이 다시 누적되고 있다
