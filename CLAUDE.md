# 마이가디언 v2 — 프로젝트 헌법

신한라이프 FC팀의 보험 업무 플랫폼. 이 문서의 원칙은 모든 작업에 우선한다.

## 언어 (최우선 규칙)

- 모든 응답·보고·커밋 메시지·문서는 한국어로 작성한다.
- 첫 문장부터 한국어다. 작업 시작 멘트, 도구 호출 사이의 짧은 진행 멘트, 계획 서술, 오류 보고까지 예외 없다.
- 하위 에이전트(Explore, Task 등)를 띄울 때도 프롬프트에 "응답은 한국어"를 명시해서 넘긴다.
- 영어는 코드·명령어·파일명·기술 고유명사에만 허용한다.

## 세션 운영

- 작업 시작 전 `docs/current-status.md`, `docs/decisions.md`, `docs/handoff.md`를 읽는다.
- 세션 종료 전 변경 내용·검증 결과·미완료 항목을 `docs/handoff.md`에 기록하고 commit·push한다.
- 도구 분업: 앱 세션 = 화면·디자인 전담 / 터미널 세션 = 데이터·파이프라인·자동화 전담. 같은 파일을 두 세션이 동시에 만지지 않는다. 작업 전환 시 반드시 commit·push.
- 협업 역할: Claude Code = 주 시공자, Codex = 검수·검증 전담. 브랜치는 main(검증 완료본) / claude/*(시공) / codex/*(검수), 머지 권한은 사용자에게.

## 아키텍처 — 순수 정적

- HTML + CSS + JS + 정적 JSON만 사용한다.
- 상시 서버·데이터베이스·로그인 기능 금지.
- 사용자 입력은 브라우저 내에서만 처리하며 외부 전송을 금지한다.
- 예외: 브리핑 실시간 동기화는 추후 Cloudflare Workers 엣지 함수로 구현한다(2차 공사, 지금은 손대지 않음).

## 디자인

- 컨셉: 금융권 최상위 업무 플랫폼의 절제된 UI. 기준 실물은 신한라이프 SOL FC — 일하는 화면의 밀도를 목표로 한다.
- 업무 화면은 정보 밀도 우선: 필터바(조건 셀렉트 + 조회) + 테이블 + 상세의 조회 화면 문법을 따른다. 홈은 랜딩이 아니라 대시보드다(실데이터 수치 타일·목록·캘린더).
- 밀도는 실데이터로만 만든다. 가짜 수치로 채운 화면 금지. 데이터가 없으면 해당 구획 자체를 만들지 않는다.
- 단, SOL FC의 메뉴 비대화(39개)는 반면교사 — 메뉴는 7개(고객관리 포함)를 유지하고 화면 안을 깊게 만든다.
- 기준 기기 우선순위: ① 태블릿(1024~1366px, 상담 중 고객과 함께 보는 화면) ② 데스크톱(1920px, 업무 주력) ③ 모바일(390px, 깨지지 않게 지원).
- 예외: `web/care/`와 `web/card/`는 모바일 퍼스트 유지 — 독자가 고객(카톡 유입)이다.
- 화면의 주인에 따라 문법을 가른다 (원칙):
  - 설계사 화면(플랫폼) = 디자인 + 실무 효율. 조회 밀도가 우선이다.
  - 고객 화면(care/card/r) = 가독성 + 잡지급 디자인 퀄리티. 기준 실물은 포브스·매경이코노미류 경제지의 편집 문법(제호·킥커·표제 위계·1단 본문 조판·인용구·헤어라인·이미지 캡션·칼럼 박스).
  - 목표 수준: 현존 업계 최고이자 앞으로도 최고 — 디자인과 가독성 모두. "이 정도면 됐다"는 없다.
- 케어센터 편집장 원칙: 발행인이 누구든 편집장은 항상 안창민이다 (표지·판권 명기, 팀원 호는 발행 OOO FC 병기).
- 팔레트: 바탕 `#ffffff`/`#fafbfc`, 텍스트 잉크 네이비 `#1a2744`, 포인트 블루 `#2563eb`(버튼·활성·링크 전용), 상태색 지급=그린·부지급=레드·일부지급=앰버.
- 장식 목적의 색상 사용 금지. 색은 신호로만 쓴다. 골드 `#c9a14a`는 카톡카드·인쇄물 전용.
- 서체: 플랫폼 UI 전체는 Pretendard 단일 서체. 위계는 서체 교체가 아니라 크기·굵기·색으로만 만든다.
- Noto Serif KR은 케어센터 아티클 제목·카톡카드·인쇄물 전용. 플랫폼 UI(사이드바·제목·버튼 등)에 세리프 사용 금지.
- 타이포 위계: 핵심 수치 32pt 이상 굵게, 섹션 제목 20px 굵게(자간 -0.02em), 본문 15px, 보조 라벨 13px 연회색.
- 한글 조판: 본문 word-break: keep-all, 제목 자간 -0.02em, 수치는 tabular-nums 고정폭.
- 내비게이션: 데스크톱·태블릿은 좌측 사이드바(홈/고객관리/청구·지급/사례·판례/학습/케어센터/카톡카드), 모바일에서만 햄버거로 축소. 전 화면 동일 메뉴. admin은 어디에도 미노출.
- 레이아웃: 검색·목록형 화면은 2단(좌 목록 + 우 상세) 구조 허용. 홈 최상단에 통합 검색창 자리.
- 여백: 섹션 간 48px 이상, 카드 패딩 24px 이상.
- 그림자는 subtle, radius 12px 통일, 그라데이션 금지. 모션은 절제된 기능 모션만: 등장·전환 0.4초 이내 감속 곡선, prefers-reduced-motion 존중, 무한 반복·장식 애니메이션 금지.
- 다크모드는 CSS 변수 구조로 작성한다 (구현은 후순위, 구조만 마련).
- 인쇄 최적화 (A4/A3 출력 시 가독성 우선)
- 스크린샷 검증 해상도: 1920px / 1180px(태블릿) / 390px 3종.

## 표기 금지

- 이모지 사용 금지
- 영문 장식 표기 금지 (STEP/PHASE는 예외)
- AI 말투 금지 (예: "~해 드릴게요", "물론입니다")
- 은유 표현 금지

## 데이터 신뢰성 — 절대 원칙

- 판례·사례를 절대 생성하지 않는다. `data/`의 실데이터만 표시하고, 없으면 "관련 사례 없음"으로 표시한다.
- 사건번호 없는 판례는 표시하지 않는다.
- 모든 사례·판례에 신뢰도 등급(검증완료/부분검증/미검증-참고용)을 필수로 표기한다.
- 도구 결과 화면에는 다음 문구를 고정 표시한다:
  "본 자료는 공개된 분쟁 사례 정보 제공이며 개별 지급 여부 판단이 아닙니다"

## 개인정보

- 고객 실명·연락처 등 개인정보는 이 저장소에 절대 저장하지 않는다.
- 고객은 코드(C-YYYY-NNN)로만 표기한다.

## v1 계승 원칙

- 데이터는 추출만 한다. 변환·합산·매핑 금지. (예: 엑셀 생성은 AI 호출 없이 저장된 데이터의 포맷 변환만)
- 한번 만든 로직에 패치 위에 패치를 쌓지 않는다. 더러워지면 다시 짓는다.
- 세션 종료 전 반드시 commit·push 한다.

## 기능 추가 브레이크

- 기능을 추가하기 전에 반드시 `docs/legacy-operations.md`의 "버려도 되는 기능" 목록과 대조한다.
- 목록에 있는 기능(태그·계약증서·상세정보·등급·파이프라인단계·게임·위임장·추천관리 등)은 다시 만들지 않는다.
